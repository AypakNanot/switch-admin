## 二、用户与权限

### 用户场景与价值

**典型用户**：系统管理员、运维人员

**核心场景**：
1. **多用户管理**：大型网络通常由多人协作管理，需要为不同角色创建不同权限级别的账号
2. **权限隔离**：防止低权限用户误操作关键配置，如只读用户不能修改路由/端口配置
3. **会话审计**：查看当前登录用户，发现异常登录或僵尸会话时强制踢出

**业务价值**：
- 支持多人协作，避免账号共享导致的安全风险
- 权限分级降低误操作风险
- 会话管理提升系统安全性

**安全约束**：
- 用户管理属于高危操作，建议仅 admin 角色可操作
- 删除用户前需确认该用户无活跃会话
- 密码需符合复杂度要求

### 2.1 用户管理

**功能说明**：用户账号管理功能，用于创建、编辑、删除系统用户，并分配权限等级。

#### 2.1.1 界面布局

**菜单路径**：维护 → 用户管理

```
┌─────────────────────────────────────────────────────────┐
│ 用户管理                                                 │
├─────────────────────────────────────────────────────────┤
│ [添加] [删除] [刷新]                                     │
├─────────────────────────────────────────────────────────┤
│ ☐ │ 用户名   │ 权限等级 │ 密码 │ 操作                   │
├─────────────────────────────────────────────────────────┤
│ ☐ │ admin1  │          │      │ 编辑                   │
└─────────────────────────────────────────────────────────┘
```

#### 2.1.2 列表字段定义

| 字段名 | 类型 | 说明 |
|--------|------|------|
| 用户名 | String | 用户账号名称，唯一标识 |
| 权限等级 | Enum | 权限级别，详见权限等级说明 |
| 密码 | String | 显示为占位符（如******），不可见明文 |
| 操作 | - | 编辑按钮 |

#### 2.1.3 权限等级定义

| 等级 | 名称 | 权限范围 |
|------|------|----------|
| 0 | super-admin | 超级管理员，唯一且不可删除，可管理所有用户（包括同级 admin） |
| 1 | admin | 普通管理员，可管理 operator/viewer，但不能管理其他 admin 和 super-admin |
| 2 | operator | 操作权限，可修改大部分配置，但不能管理用户、重启设备 |
| 3 | viewer | 只读权限，仅可查看状态，不可修改任何配置 |

**Super Admin 说明**：

| 项目 | 说明 |
|------|------|
| 默认账号 | `admin`，系统初始化时自动创建 |
| 不可删除 | 不能被任何用户删除（包括自己） |
| 不可降级 | 不能被修改为更低权限等级 |
| 唯一性 | 系统只能有一个 super-admin |

**权限越权防御规则**：

| 操作 | super-admin | admin | 说明 |
|------|-------------|-------|------|
| 踢出 super-admin | ❌ 禁止 | ❌ 禁止 | 超级管理员不能被踢出 |
| 删除 super-admin | ❌ 禁止 | ❌ 禁止 | 超级管理员不能被删除 |
| 踢出其他 admin | ✅ 允许 | ❌ 禁止 | 普通管理员之间不能互相踢出 |
| 删除其他 admin | ✅ 允许 | ❌ 禁止 | 普通管理员之间不能互相删除 |
| 踢出 operator/viewer | ✅ 允许 | ✅ 允许 | 可以踢出下级用户 |
| 删除 operator/viewer | ✅ 允许 | ✅ 允许 | 可以删除下级用户 |

**设计理由**：
- 防止同级管理员互相攻击（如 admin1 踢出 admin2、删除 admin2 账号）
- Super Admin 作为"最后的防线"，确保系统永远有一个可管理账号

**权限与按钮状态矩阵**：

| 功能模块 | admin | operator | viewer |
|----------|-------|----------|--------|
| 添加用户 | ✅ | ❌ (置灰) | ❌ (隐藏) |
| 删除用户 | ✅ | ❌ (置灰) | ❌ (隐藏) |
| 编辑用户 | ✅ | ❌ (置灰) | ❌ (隐藏) |
| 查看 用户列表 | ✅ | ✅ | ✅ |

#### 2.1.4 添加/编辑用户表单

**界面布局**：

```
┌─────────────────────────────────────────────────────────┐
│ 添加/修改用户                                            │
├─────────────────────────────────────────────────────────┤
│ 用户名      [________________]                          │
│             (以字母开头，只能包含[0-9a-zA-Z-]，长度1-32) │
│ 密码        [________________]                          │
│ 确认密码    [________________]                          │
│ 权限等级    [▼ 请选择        ]                          │
├─────────────────────────────────────────────────────────┤
│                          [返回]                         │
└─────────────────────────────────────────────────────────┘
```

#### 2.1.5 表单字段定义

| 字段 | 类型 | 必填 | 校验规则 | 说明 |
|------|------|------|----------|------|
| 用户名 | String | 是 | 以字母开头，仅含 [0-9a-zA-Z-]，长度 1-32 | 全局唯一，不可重复 |
| 密码 | String | 是 | 长度 8-32，需包含大小写字母和数字 | 编辑时为空表示不修改密码 |
| 确认密码 | String | 是 | 必须与密码一致 | 防止输入错误 |
| 权限等级 | Enum | 是 | 1/2/3 | admin/operator/viewer |

#### 2.1.6 输入校验规则（前端实时校验）

**用户名校验**（失焦触发）：

| 校验项 | 规则 | 错误提示 |
|--------|------|----------|
| 必填校验 | 不能为空 | "请输入用户名" |
| 首字符 | 必须为字母 | "用户名必须以字母开头" |
| 字符范围 | 仅允许 [0-9a-zA-Z-] | "用户名只能包含字母、数字和中划线" |
| 长度范围 | 1-32 字符 | "用户名长度为1-32个字符" |
| 唯一性 | 不能与已有用户名重复（添加时） | "用户名已存在" |

**密码校验**（失焦触发）：

| 校验项 | 规则 | 错误提示 |
|--------|------|----------|
| 必填校验 | 不能为空（添加时必填，编辑时可为空） | "请输入密码" |
| 长度范围 | 8-32 字符 | "密码长度为8-32个字符" |
| 复杂度 | 必须包含大写字母、小写字母、数字 | "密码必须包含大小写字母和数字" |
| 禁用弱密码 | 不能是 "admin123"、"password" 等常见弱密码 | "密码过于简单，请使用更复杂的密码" |

**确认密码校验**：

| 校验项 | 规则 | 错误提示 |
|--------|------|----------|
| 一致性 | 必须与密码字段完全一致 | "两次输入的密码不一致" |

#### 2.1.7 交互逻辑

**主流程（Happy Path）**：

| 操作 | 流程 |
|------|------|
| 添加用户 | 点击「添加」→ 弹出表单 → 填写信息 → 确认保存 → 刷新列表 |
| 编辑用户 | 点击「编辑」→ 弹出表单（用户名不可修改，密码为空表示不改）→ 修改 → 保存 |
| 删除用户 | 勾选用户 → 点击「删除」→ 二次确认 → 确认删除 |

**异常流程（Exception Flow）**：

| 异常场景 | 处理方式 |
|----------|----------|
| 用户名已存在 | 提示"用户名已存在，请使用其他用户名" |
| 删除当前登录用户 | 提示"不能删除当前登录的用户" |
| 删除最后一个 admin 用户 | 提示"系统至少需要保留一个管理员账号，无法删除" |
| 编辑用户时用户已被删除 | 提示"用户不存在，请刷新列表" |
| **删除 super-admin** | 提示"超级管理员账号不可删除" |
| **删除同级 admin（非 super-admin）** | 提示"权限不足，不能删除同级管理员账号" |
| **踢出 super-admin 会话** | 提示"超级管理员会话不可被踢出" |
| **踢出同级 admin 会话（非 super-admin）** | 提示"权限不足，不能踢出同级管理员会话" |

**安全措施**：

| 措施 | 说明 |
|------|------|
| 密码不明文显示 | 列表中密码列显示为占位符 |
| 编辑时密码可选 | 编辑表单密码为空表示不修改密码，不为空则更新密码 |
| 操作日志 | 用户添加/修改/删除操作需记录到系统日志 |

#### 2.1.8 API 结构设计

```
GET    /api/v1/users                    # 获取用户列表
POST   /api/v1/users                    # 创建用户
GET    /api/v1/users/{username}         # 获取用户详情
PUT    /api/v1/users/{username}         # 更新用户信息
DELETE /api/v1/users                    # 批量删除用户
DELETE /api/v1/users/{username}         # 删除单个用户
```

**创建用户请求**：

```json
// POST /api/v1/users
{
  "username": "operator1",
  "password": "Secure@123",
  "role": 2
}
```

**成功响应**：

```json
{
  "code": 200,
  "message": "用户创建成功",
  "data": {
    "username": "operator1",
    "role": 2,
    "created_at": "2025-03-12 14:00:00"
  }
}
```

**错误响应**：

```json
// 用户名已存在
{
  "code": 409,
  "error": "USER_EXISTS",
  "message": "用户名已存在，请使用其他用户名"
}

// 不能删除最后一个管理员
{
  "code": 400,
  "error": "LAST_ADMIN",
  "message": "系统至少需要保留一个管理员账号，无法删除"
}
```

### 2.2 当前会话

**功能说明**：会话管理功能，用于查看当前所有登录用户的会话信息，支持强制踢出异常会话。

#### 2.2.1 界面布局

**菜单路径**：维护 → 当前会话

```
┌─────────────────────────────────────────────────────────┐
│ 当前会话                                                 │
├─────────────────────────────────────────────────────────┤
│ [删除] [刷新]                                            │
├─────────────────────────────────────────────────────────┤
│ ☐ │ 用户名  │ 会话ID      │ 超时时间           │ 客户端IP   │
├─────────────────────────────────────────────────────────┤
│ ☐ │ admin1 │ 1773322174  │ 2026-03-12 14:01:14│ 213.213.9.239 (*) │
└─────────────────────────────────────────────────────────┘
```

#### 2.2.2 字段定义

| 字段名 | 类型 | 说明 |
|--------|------|------|
| 用户名 | String | 登录用户名 |
| 会话ID | String | 会话唯一标识 |
| 超时时间 | DateTime | 会话过期时间，超时后自动登出 |
| 客户端IP | String | 登录来源 IP 地址，(*) 表示当前会话 |

#### 2.2.3 交互逻辑

**主流程**：

| 操作 | 流程 |
|------|------|
| 查看会话 | 进入页面自动加载当前所有活跃会话 |
| 刷新会话 | 点击「刷新」重新加载会话列表 |
| 强制踢出 | 勾选会话 → 点击「删除」→ 二次确认 → 强制登出该用户 |

**异常流程**：

| 异常场景 | 处理方式 |
|----------|----------|
| 踢出当前会话 | 提示"确定要登出当前登录账号吗？操作后需要重新登录" → 确认后登出 |
| 会话已过期 | 刷新时自动过滤已过期会话 |

**被踢用户的被动登出体验（关键！）**：

```
当管理员强制踢出某个用户的会话时，被踢用户的前端必须做出响应：

方案一：WebSocket 实时通知（推荐）
• 后端通过 WebSocket 向被踢用户推送 { "event": "force_logout", "message": "您的账号已被管理员强制下线" }
• 前端收到消息后立即弹出全局提示框

方案二：API 拦截（兜底方案）
• 被踢用户下次发起任何 API 请求时，后端返回 401 + 特定错误码
• 前端拦截器检测到该错误码后弹出全局提示框

被踢用户看到的界面：
┌─────────────────────────────────────────────────────────┐
│                    ⚠️ 账号已下线                         │
├─────────────────────────────────────────────────────────┤
│         您的账号已被管理员强制下线                       │
│                                                          │
│         如有疑问，请联系系统管理员。                     │
│                                                          │
│                              [确定，重新登录]            │
└─────────────────────────────────────────────────────────┘

点击确定后，跳转至登录页面，清除本地所有会话状态。

禁止行为：
• ❌ 仅在控制台抛出 401 错误而页面无反应
• ❌ 页面卡死无任何提示
• ❌ 显示技术性错误信息（如 "Unauthorized"）
```

#### 2.2.4 API 结构设计

```
GET    /api/v1/sessions                 # 获取会话列表
DELETE /api/v1/sessions/{session_id}    # 强制终止会话
```

## 三、系统管理

### 用户场景与价值

**典型用户**：系统管理员

**核心场景**：
1. **配置持久化**：修改配置后需要保存，否则重启后丢失
2. **固件升级**：通过文件管理上传新版本固件并升级
3. **故障恢复**：设备配置损坏时，从备份配置恢复
4. **设备维护**：远程重启设备、恢复出厂设置

**业务价值**：
- 确保配置不丢失
- 支持远程运维，减少现场操作
- 提供故障恢复能力

**风险提示**：
- 重启/恢复出厂配置属于高危操作，可能导致业务中断
- 需要二次确认机制

---

### 3.1 重启/保存

**功能说明**：设备运维操作功能，包括保存配置、重启设备、恢复出厂配置。

#### 3.1.1 界面布局

**菜单路径**：维护 → 重启/保存

```
┌─────────────────────────────────────────────────────────┐
│ 保存配置                                                 │
├─────────────────────────────────────────────────────────┤
│                     [保存配置]                          │
├─────────────────────────────────────────────────────────┤
│ 重启交换机                                               │
├─────────────────────────────────────────────────────────┤
│ ☐ 重启之前自动保存配置                                   │
│                     [重启交换机]                        │
├─────────────────────────────────────────────────────────┤
│ 恢复出厂配置                                             │
├─────────────────────────────────────────────────────────┤
│                     [恢复]                              │
└─────────────────────────────────────────────────────────┘
```

#### 3.1.2 功能详细说明

| 功能 | 说明 | 风险等级 | 权限要求 |
|------|------|----------|----------|
| 保存配置 | 将当前运行配置写入启动配置文件，重启后生效 | 低 | admin/operator |
| 重启交换机 | 立即重启设备，业务将中断 1-3 分钟 | 高 | admin |
| 恢复出厂配置 | 清除所有配置，恢复到出厂默认状态 | 极高 | admin |

#### 3.1.3 交互逻辑

**保存配置**：

```
1. 点击「保存配置」
2. 按钮变为"保存中..."
3. 后端执行配置保存
4. 成功 → 提示"配置保存成功"
5. 失败 → 提示"配置保存失败，请稍后重试"
```

**重启交换机**：

```
1. 点击「重启交换机」
2. 弹出二次确认框：
   "确定要重启交换机吗？重启期间网络将中断约1-3分钟。"
   [取消] [确认重启]
3. 若勾选"重启之前自动保存配置"→ 先保存再重启
4. 确认后执行重启
5. 前端显示"设备正在重启，请稍后刷新页面..."
6. 建议倒计时 60 秒后自动刷新页面
```

**恢复出厂配置**：

```
1. 点击「恢复」
2. 弹出高危确认框：
   "⚠️ 警告：此操作将清除所有配置并恢复出厂默认状态！
   
   当前配置将永久丢失，且无法恢复。
   
   重启后，设备管理 IP 将恢复为默认地址：192.168.1.1
   请确保您知道如何访问默认地址。
   
   请输入 'CONFIRM' 以确认此操作："
   [____________]
   [取消] [确认恢复]
3. 用户必须手动输入 'CONFIRM' 才能点击确认
4. 确认后执行恢复出厂配置
5. 显示结果提示（见下方"恢复出厂配置后的用户指引"）
```

**恢复出厂配置后的用户指引（关键！）**：

```
执行成功后，前端显示全屏提示页面（不自动刷新！）：

┌─────────────────────────────────────────────────────────┐
│ ⚠️ 设备正在恢复出厂设置并重启                           │
├─────────────────────────────────────────────────────────┤
│ 重启后，设备将恢复为出厂默认配置：                       │
│                                                          │
│ • 管理 IP：192.168.1.1                                   │
│ • 子网掩码：255.255.255.0                                │
│ • 默认用户名/密码：admin/admin                           │
│                                                          │
│ 请按以下步骤重新访问设备：                               │
│ 1. 将您的电脑 IP 修改为 192.168.1.x（如 192.168.1.100） │
│ 2. 等待设备重启完成（约 2-3 分钟）                       │
│ 3. 在浏览器中访问 http://192.168.1.1                    │
│                                                          │
│ ⚠️ 当前页面将无法自动刷新，请勿点击刷新按钮              │
│                                                          │
│                                    [我已知晓，关闭页面]  │
└─────────────────────────────────────────────────────────┘

设计理由：
- 恢复出厂后管理 IP 变更，当前页面 URL 已失效
- 自动刷新会导致浏览器 ERR_CONNECTION_TIMED_OUT
- 用户可能误以为设备损坏
- 必须明确告知新的访问方式
```

#### 3.1.4 异常流程

| 异常场景 | 处理方式 |
|----------|----------|
| 保存配置时配置未变更 | 提示"配置未变更，无需保存" |

**发送测试告警（Test Trap）**：

```
点击「测试」按钮后：

1. 弹出确认框：
   "确定要发送测试 Trap 吗？
   
   目标地址: 192.168.1.100
   UDP 端口: 162
   
   测试 Trap 将发送到目标主机，请检查网管平台是否收到。"
   [取消] [发送测试]

2. 后端发送测试 Trap（OID: 1.3.6.1.6.3.1.1.5.999，描述："Test Trap from Switch"）

3. 前端显示结果：
   成功 → 提示"测试 Trap 已发送，请检查网管平台是否收到告警"
   失败 → 提示"测试 Trap 发送失败，请检查网络连通性和目标主机配置"
```

**设计理由**：
- 网管配完 Trap 目标后，最头疼的是"不知道配对了没有"
- 以前只能真的去拔一根网线来触发 Linkdown Trap，或等设备真的出问题
- 测试按钮让网管当场验证配置正确性，省去无数沟通成本

```

```

## 五、安全防护

### 用户场景与价值

**典型用户**：网络安全管理员

**核心场景**：
1. **蠕虫防护**：防御已知蠕虫病毒攻击（如 SQL Slammer、Sasser 等）
2. **DDoS防护**：防御分布式拒绝服务攻击（ICMP/UDP/SYN Flood）
3. **ARP防护**：防御 ARP 欺骗攻击，防止网关被劫持

**业务价值**：
- 提供基础网络安全防护能力
- 无需额外安全设备即可防御常见攻击
- 降低网络安全风险

**注意事项**：
- 防护功能会消耗设备 CPU 资源
- 过于严格的阈值可能误杀正常流量
- 需要根据实际网络环境调优
