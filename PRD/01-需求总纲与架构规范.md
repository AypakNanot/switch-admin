## 一、来源 (Source) - 溯源与价值

### 1.1 产品定位

基于 Web 的智能交换机管理系统，为网络运维人员提供可视化的设备配置、监控、维护和诊断能力，降低命令行操作门槛，提升运维效率。

### 1.2 目标用户

| 用户角色 | 描述 | 核心诉求 |
|----------|------|----------|
| **网络运维工程师** | 日常负责网络设备配置、监控、故障排查 | 快速定位故障、可视化配置、批量操作 |
| **系统管理员** | 负责设备用户管理、系统维护、安全策略 | 权限控制、配置备份、安全防护 |
| **网络规划工程师** | 负责网络拓扑规划、VLAN 划分、路由设计 | 拓扑可视化、配置模板、批量部署 |

### 1.3 业务价值

| 价值维度 | 说明 |
|----------|------|
| **降低运维门槛** | 可视化界面替代命令行，减少误操作风险 |
| **缩短故障 MTTR** | 实时监控 + 诊断工具，快速定位故障点 |
| **提升配置效率** | 批量操作 + 配置校验，减少重复劳动 |
| **保障网络安全** | 权限分级 + 安全防护，防止未授权访问和攻击 |

### 1.4 运行环境约束（关键！）

| 约束项 | 规格 | 影响 |
|--------|------|------|
| **设备内存** | 256MB ~ 512MB | 严禁大数据量内存驻留，CSV 导出需流式处理 |
| **CPU** | 单核 ~ 双核 ARM/MIPS | 限制并发处理能力，需防 DDoS 式刷新 |
| **存储** | 128MB ~ 512MB Flash | 历史数据存储受限，禁止频繁写入 Flash |
| **网络带宽** | 10M/100M/1000M/10G | 计数器需 64 位防溢出（10G 端口 3.4 秒溢出 32 位） |

## 三、规格 (Specs) - 工程化契约

### 3.1 公共界面交互规范

#### 3.1.1 表单输入校验规则

| 校验类型 | 规则 | 错误提示模板 | 触发时机 |
|----------|------|--------------|----------|
| **必填校验** | 不能为空 | "请输入{字段名}" | 失焦/提交 |
| **长度校验** | min~max 字符 | "{字段名}长度为{min}-{max}个字符" | 失焦 |
| **格式校验** | 正则匹配 | "请输入有效的{字段名}格式" | 失焦 |
| **范围校验** | min~max 数值 | "{字段名}范围为{min}-{max}" | 失焦 |
| **唯一性校验** | 不能重复 | "{字段名}已存在" | 失焦/提交 |
| **一致性校验** | 两字段一致 | "两次输入的{字段名}不一致" | 失焦 |

#### 3.1.2 特殊字段校验规则

| 字段类型 | 校验规则 | 错误提示 |
|----------|----------|----------|
| **IPv4 地址** | 合法 IPv4 格式，不能是广播/组播/环回 | "请输入有效的 IPv4 地址" |
| **MAC 地址** | 格式 xx:xx:xx:xx:xx:xx，不能是全 0/全 F | "请输入有效的 MAC 地址" |
| **端口描述** | ≤64 字符，支持中文、英文、数字、下划线、横杠 | "描述只能包含中文、英文、数字、下划线和横杠" |
| **VLAN ID** | 范围 1-4094，1 为默认 VLAN，4095 保留 | "VLAN ID 范围为 1-4094" |
| **团体名称** | 1-128 字符，不能包含 ? > " \ | "团体名称不能包含特殊字符" |
| **用户名** | 字母开头，仅含 [0-9a-zA-Z-]，1-32 字符 | "用户名格式不正确" |
| **密码** | 8-32 字符，需包含大小写字母和数字 | "密码必须包含大小写字母和数字" |

#### 3.1.3 按钮状态规范

| 状态 | 行为 | 视觉反馈 |
|------|------|----------|
| **正常** | 可点击 | 标准样式 |
| **Loading** | 禁用点击，显示进度 | 按钮文字变为"保存中..."，显示 spinner |
| **置灰** | 禁用点击，显示提示 | 灰色样式，hover 显示"无权限"提示 |
| **隐藏** | 不渲染 | 根据权限控制不显示 |

#### 3.2.3 ⚠️ 64 位计数器约束（关键！）

```
┌─────────────────────────────────────────────────────────────────────────────┐
│ ⚠️ 工程约束：流量统计字段必须使用 64 位无符号整数                           │
├─────────────────────────────────────────────────────────────────────────────┤
│ 背景：10Gbps 端口全速传输，3.4 秒即可传输超过 42 亿字节（32 位溢出）        │
│                                                                              │
│ 强制要求：                                                                   │
│ ✅ 所有流量统计字段必须使用 uint64_t（C 语言）/ BIGINT UNSIGNED（数据库）  │
│ ✅ API 传输时超过 JavaScript 安全整数范围的数值必须作为 String 类型传输    │
│ ✅ 前端使用 BigInt 解析大数字符串                                            │
│                                                                              │
│ 受影响字段：                                                                 │
│ • rx_bytes / tx_bytes（接收/发送字节数）                                    │
│ • rx_packets / tx_packets（接收/发送包数）                                  │
│ • rx_errors / tx_errors（接收/发送错误数）                                  │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 3.3 异常流处理（至少 3 种）

#### 3.3.1 网络超时

| 场景 | 检测条件 | 处理方式 | 用户提示 |
|------|----------|----------|----------|
| **API 请求超时** | >30 秒无响应 | 前端自动重试 1 次，仍失败则报错 | "请求超时，请检查网络连接后重试" |
| **文件上传超时** | >600 秒无响应 | 支持断点续传，提示用户重试 | "上传超时，请检查网络后重试" |
| **诊断任务超时** | Ping>30s / Traceroute>120s | 后端自动终止任务，释放资源 | "诊断超时，请稍后重试" |

#### 3.3.2 并发冲突

| 场景 | 检测逻辑 | 处理方式 | 用户提示 |
|------|----------|----------|----------|
| **配置被其他用户修改** | 后端检测配置版本号变化 | 返回 409 + CONCURRENT_MODIFY 错误码 | "配置已被其他用户修改，请刷新后重试" |
| **删除已被删除的资源** | 资源不存在 | 返回 404，前端自动刷新列表 | "资源不存在，列表已刷新" |
| **重复创建同名资源** | 唯一性约束冲突 | 返回 409 + USER_EXISTS/ROUTE_EXISTS | "{资源名}已存在，请勿重复配置" |

#### 3.3.3 依赖不可达

| 场景 | 检测逻辑 | 处理方式 | 用户提示 |
|------|----------|----------|----------|
| **静态路由下一跳不可达** | 下一跳不在直连网段 | **允许保存**，列表显示⚠️警告图标 | "下一跳当前不可达，该路由暂未生效" |
| **SNMP Trap 目标不可达** | 发送测试 Trap 失败 | 提示用户检查网络连通性 | "测试 Trap 发送失败，请检查目标主机配置" |
| **固件服务器不可达** | 在线升级时连接失败 | 提示用户检查网络或稍后重试 | "无法连接升级服务器，请检查网络后重试" |

### 3.4 异步任务架构

#### 3.4.1 适用场景

| 场景 | 原因 | 设计方案 |
|------|------|----------|
| **Ping 诊断** | 可能耗时较长（网络慢） | 异步任务 + 轮询结果 |
| **Traceroute 诊断** | 可能耗时 1-2 分钟（30 跳） | 异步任务 + 轮询结果 |
| **虚拟电缆检测** | 需等待硬件检测完成 | 异步任务 + 轮询结果 |
| **固件升级** | 写入 Flash 需 2-5 分钟 | 异步任务 + 全屏遮罩 |
| **大文件上传** | 可能超过 300MB | 分片上传 + 断点续传 |

#### 3.4.2 任务状态机

```
创建任务 → running → 轮询查询 → completed/failed
                    ↓
                用户离开页面 → DELETE 终止任务
```

#### 3.4.3 ⚠️ 防孤儿进程机制（关键！）

```
┌─────────────────────────────────────────────────────────────────────────────┐
│ ⚠️ 工程约束：异步任务必须防止孤儿进程占用交换机资源                        │
├─────────────────────────────────────────────────────────────────────────────┤
│ 背景：用户执行 Ping/Traceroute 后直接关闭页面，任务仍在后台运行            │
│                                                                              │
│ 强制要求：                                                                   │
│ ✅ 前端：页面离开时（beforeUnmount）自动调用 DELETE 终止任务               │
│ ✅ 后端：定时任务（每 60 秒）扫描孤儿任务并清理                             │
│ ✅ 后端：任务创建时记录 WebSocket/SSE 连接状态，连接断开则终止任务         │
│ ✅ 后端：任务执行超时自动终止（Ping 30s / Traceroute 120s）                │
│                                                                              │
│ 数据结构示例：                                                               │
│ {                                                                           │
│   "task_id": "ping_20250312_001",                                           │
│   "created_at": "2025-03-12T15:30:00Z",                                     │
│   "websocket_connected": true,                                              │
│   "timeout_seconds": 30                                                     │
│ }                                                                           │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 3.5 资源清理机制

#### 3.5.1 统计清零偏移量机制（关键！）

```
┌─────────────────────────────────────────────────────────────────────────────┐
│ ⚠️ 工程约束：统计清零必须采用偏移量计算，禁止直接清零硬件计数器            │
├─────────────────────────────────────────────────────────────────────────────┤
│ 背景：交换芯片的硬件计数器是全局共享的，Web 清零会影响 SNMP 平台数据        │
│                                                                              │
│ 强制要求：                                                                   │
│ ✅ 采用"偏移量（Offset）机制"：记录清零时刻的计数值                         │
│ ✅ 展示时用"当前硬件值 - Offset"显示相对值                                  │
│ ✅ Offset 存储在 /var/run/port_stats_offset.json（重启清空）               │
│ ❌ 禁止写入硬件寄存器清零 ASIC 计数器                                       │
│ ❌ 禁止存储在 Flash（有写入次数限制）                                       │
└─────────────────────────────────────────────────────────────────────────────┘
```

#### 3.5.2 配置挂起与恢复机制

| 场景 | 处理方式 |
|------|----------|
| **端口加入聚合组** | 独立配置（VLAN、流控、风暴控制）挂起保存，继承聚合组配置 |
| **端口移出聚合组** | 恢复加入前的独立配置，若无配置则恢复出厂默认 |
| **堆叠成员变更** | 配置需重启生效，提前提示用户 |

### 4.2 性能与资源瓶颈防范

#### 4.2.1 防 DDoS 刷新机制（关键！）

```
┌─────────────────────────────────────────────────────────────────────────────┐
│ ⚠️ 架构级约束：前端后台休眠与后端聚合缓存                                   │
├─────────────────────────────────────────────────────────────────────────────┤
│ 危险场景：管理员开启 10 秒自动刷新后切换到其他页签，前端仍在疯狂轮询        │
│ 后果：交换机 CPU 被无效 Web 轮询消耗，导致 CLI 变卡！                       │
│                                                                              │
│ 强制防护规范：                                                               │
│                                                                              │
│ 【前端：Visibility API 拦截】                                               │
│ ✅ 当 document.hidden === true 时，自动暂停倒计时和轮询请求                │
│ ✅ 切回激活状态时，立即恢复刷新并触发一次数据拉取                           │
│                                                                              │
│ 【后端：数据聚合守护进程】                                                   │
│ ✅ 维护独立守护进程，定时（每 5 秒）异步收集数据并放入共享内存              │
│ ✅ Web API 直接读取内存响应，耗时控制在 50ms 以内                           │
│ ✅ 禁止后端收到 1 个 HTTP 请求后，向底层驱动发起几十次 CLI/内核调用         │
│                                                                              │
│ 性能指标：                                                                   │
│ • API 响应时间：≤ 100ms（读取内存）                                        │
│ • 底层采集频率：每 5 秒一次（独立于 Web 请求）                              │
│ • 内存占用：约 10KB（共享内存数据结构）                                    │
└─────────────────────────────────────────────────────────────────────────────┘
```

#### 4.2.2 防重复提交

| 措施 | 说明 |
|------|------|
| **按钮 Loading** | 点击后立即禁用按钮，显示"保存中..." |
| **请求幂等** | 后端通过 Request-ID 去重，相同请求直接返回上次结果 |
| **超时保护** | 按钮 Loading 状态最长 30 秒，超时自动恢复 |

#### 4.3.1 配置互斥约束总表

```
┌─────────────────────────────────────────────────────────────────────────────────────────────┐
│ ⚠️ 配置互斥约束一览表                                                                       │
├─────────────────────────────────────────────────────────────────────────────────────────────┤
│ 端口模式互斥：                                                                               │
│ ├─ Access 模式：必须设置 PVID，不能设置 Allowed VLAN，不能设置 Tagged VLAN                 │
│ ├─ Trunk 模式：必须设置 PVID 和 Allowed VLAN，可设置 Tagged VLAN                           │
│ └─ Hybrid 模式：可灵活设置 PVID、Untagged VLAN、Tagged VLAN                                │
│                                                                                             │
│ 功能互斥：                                                                                   │
│ ├─ 端口加入聚合组后，不能单独配置速率/双工/流控（由聚合组统一管理）                         │
│ ├─ 端口开启端口安全后，不能配置静态 MAC 地址（由安全学习机制管理）                          │
│ ├─ 端口配置 802.1X 认证后，端口安全功能受限                                                 │
│ ├─ 端口配置 ERPS 环网后，生成树协议在该端口自动禁用                                         │
│ └─ 端口配置为镜像源端口后，不能同时作为镜像目的端口                                         │
│                                                                                             │
│ 参数约束：                                                                                   │
│ ├─ PVID 必须在端口允许的 VLAN 列表中                                                        │
│ ├─ Allowed VLAN 不能包含保留 VLAN（1, 4095）                                                │
│ ├─ 聚合组 ID 范围：1-128（部分设备 1-64）                                                   │
│ ├─ 生成树优先级必须为 4096 的倍数（0, 4096, 8192, ... 61440）                               │
│ └─ MAC 地址学习上限：0-4096（0 表示不限制）                                                 │
└─────────────────────────────────────────────────────────────────────────────────────────────┘
```

#### 4.3.2 四大"毁网级"陷阱防护机制

| 陷阱名称 | 风险场景 | 防护机制 |
|----------|----------|----------|
| **自杀式切断防护** | 修改管理端口 VLAN 导致自己断网 | 红色警告 + 双重确认 + 30 秒倒计时 |
| **聚合继承法则** | 端口加入聚合组后配置混乱 | 独立配置挂起保存，移出后恢复 |
| **PoE 实际状态** | 管理启用但实际未供电（功率不足） | 区分管理状态与运行状态，显示 Power Denied 及原因 |
| **堆叠分裂惩罚** | 双主冲突导致 MAC 风暴和 IP 冲突 | 备机业务端口强制 Error-Down |

#### 4.3.3 高危操作确认机制

| 操作 | 确认级别 | 确认方式 |
|------|----------|----------|
| **重启设备** | 中级 | 二次确认弹窗 |
| **恢复出厂配置** | 高级 | 输入"CONFIRM"确认 |
| **修改管理 IP** | 中级 | 二次确认 + 警告新 IP |
| **ARP 防护设为 0** | 高级 | 输入"I UNDERSTAND"确认 + 警告后果 |
| **全局统计清零** | 高级 | 输入"CLEAR ALL"确认 |
| **固件降级** | 中级 | 二次确认 + 降级警告 |

## 五、大师的黑客视角 - 致命漏洞与破局建议

### 🔴 漏洞 #1：配置回滚机制缺失

**问题描述**：
当前需求文档中，所有配置操作都是"应用即生效"，没有配置回滚机制。一旦管理员误操作（如错误配置 VLAN 导致全网中断），只能手动恢复或重启设备，MTTR（平均修复时间）过长。

**影响范围**：
- 所有配置模块（VLAN、路由、端口配置等）
- 特别是远程运维场景，误操作后可能失去设备管理权限

**真实场景**：
> 网管小王凌晨 2 点远程配置核心交换机 VLAN，误将管理 VLAN 从 Trunk 的 Allowed VLAN 中移除，立即失去连接。由于没有配置回滚机制，只能通知机房值班人员现场 Console 恢复，业务中断 45 分钟。

**破局建议**：

```
┌─────────────────────────────────────────────────────────────────────────────┐
│ 解决方案：配置变更事务与自动回滚机制                                        │
├─────────────────────────────────────────────────────────────────────────────┤
│ 1. 【配置快照】                                                            │
│    每次配置变更前，自动创建配置快照（flash:/config-backup-{timestamp}.cfg）│
│                                                                              │
│ 2. 【变更事务】                                                            │
│    高危操作（VLAN、路由、管理 IP）采用事务模式：                            │
│    • 开始事务 → 应用配置 → 验证连通性 → 提交/回滚                          │
│                                                                              │
│ 3. 【连通性验证】                                                          │
│    配置应用后，自动检测与管理站的连通性：                                   │
│    • 成功：提交配置，提示"配置已应用"                                      │
│    • 失败：等待 60 秒 → 仍失败则自动回滚到快照，发送告警邮件                │
│                                                                              │
│ 4. 【一键回滚 UI】                                                         │
│    维护 → 配置历史 → 显示最近 10 次配置快照 → 点击"回滚" → 二次确认         │
│                                                                              │
│ 技术实现：                                                                   │
│ • 后端：配置变更前调用 save_config 备份，记录操作日志                       │
│ • 前端：高危操作页面增加"配置历史"入口                                      │
│ • 底层：支持配置回滚命令（如 configure replace）                            │
└─────────────────────────────────────────────────────────────────────────────┘
```

---

### 🔴 漏洞 #2：多用户并发配置冲突检测不足

**问题描述**：
当前需求文档中，虽然提到了"并发修改冲突"返回 409 错误，但缺乏细粒度的配置锁机制。多个管理员可能同时修改同一配置项（如同时修改 VLAN 10 的成员端口），导致配置覆盖或冲突。

**影响范围**：
- 所有配置模块，特别是多人协作场景
- VLAN 配置、端口配置、路由配置等高频修改项

**真实场景**：
> 网管 A 和网管 B 同时编辑 VLAN 10 配置。A 添加端口 1-10，B 添加端口 11-20。B 先保存，A 后保存，结果 B 的配置被 A 覆盖，端口 11-20 丢失。

**破局建议**：

```
┌─────────────────────────────────────────────────────────────────────────────┐
│ 解决方案：细粒度配置锁与乐观锁机制                                          │
├─────────────────────────────────────────────────────────────────────────────┤
│ 1. 【配置项版本号】                                                        │
│    每个配置项增加 version 字段，每次修改 version+1                          │
│                                                                              │
│ 2. 【乐观锁检测】                                                          │
│    提交配置时携带 version，后端比对：                                       │
│    • version 匹配 → 应用配置，version+1                                    │
│    • version 不匹配 → 返回 409 + 当前配置，提示"配置已被他人修改"          │
│                                                                              │
│ 3. 【配置编辑锁（可选）】                                                  │
│    用户打开编辑表单时，获取编辑锁（300 秒有效期）：                         │
│    • 其他用户看到"XXX 正在编辑此配置"，只能查看不能编辑                    │
│    • 锁超时自动释放                                                         │
│                                                                              │
│ 4. 【冲突解决 UI】                                                         │
│    检测到冲突时，显示配置对比界面：                                         │
│    ┌─────────────────────────────────────────────────────────────────┐     │
│    │ 当前配置（已被 XXX 修改）    │ 你的修改                          │     │
│    │ ├─ 端口：1-20               │ ├─ 端口：1-10                    │     │
│    │ └─ 名称：Server             │ └─ 名称：Server                  │     │
│    │                                                                    │     │
│    │ [放弃我的修改]  [查看差异并合并]  [强制覆盖（需确认）]           │     │
│    └─────────────────────────────────────────────────────────────────┘     │
│                                                                              │
│ 技术实现：                                                                   │
│ • 后端：配置表增加 version 字段，更新时 WHERE version = {old_version}      │
│ • 前端：编辑表单加载时记录 version，提交时携带                              │
│ • Redis：存储编辑锁（key: config_lock:{resource_id}, TTL: 300s）           │
└─────────────────────────────────────────────────────────────────────────────┘
```
